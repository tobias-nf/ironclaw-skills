# Bug Report — IronClaw Developer Feedback

When the user encounters a problem and wants to report it, create a structured markdown bug report with the problem summary and relevant chat history, saved to disk.

## When to use

- User says "report a bug", "file a bug", "something broke"
- User says "create a bug report", "feedback for the devs"
- User describes a problem and asks to document it

## Workflow

### Step 1: Gather the problem

Ask the user (only if not already clear from context):
1. **What happened?** — brief description of the issue
2. **What did you expect?** — expected behavior
3. **Which skill was involved?** — if applicable (tasks, meetings, email-to-tasks, etc.)

If the problem is already clear from the conversation, don't ask — extract it directly.

### Step 2: Collect context

Gather automatically (do not ask):
- **Agent version:** check IronClaw version if available
- **Skill versions:** read `version:` from relevant SKILL.md files via `memory_read`
- **Current channel/platform:** from the conversation context
- **Timestamp:** current date and time

### Step 3: Extract relevant chat history

Look back through the recent conversation for:
- The messages where the problem occurred or was described
- Any error messages, tool call failures, or unexpected responses
- The user's attempts to work around the issue

Include the relevant messages verbatim — do not summarize them. Developers need the exact sequence to reproduce.

**Rules for chat history:**
- Include only the relevant portion, not the entire conversation
- Preserve exact tool calls, responses, and error messages
- Mark each message with role (user/assistant) and timestamp if available
- Redact obvious secrets (API keys, passwords) — replace with `[REDACTED]`
- Keep it under 200 messages — if longer, include the start and end with a note about omitted middle

### Step 4: Write the report

Save to disk:
```
file_write(path="bug-reports/<date>-<slug>.md", content="...")
```

Filename: `bug-reports/YYYY-MM-DD-<short-slug>.md`

**Report template:**

```markdown
# Bug Report: <short title>

**Date:** YYYY-MM-DD HH:MM
**Reporter:** <user's agent-id or name>
**Skill:** <skill name and version, or "core" if not skill-specific>
**Platform:** <channel/platform>

## Problem

<2-4 sentence description of what went wrong>

## Expected Behavior

<What should have happened>

## Steps to Reproduce

1. <step>
2. <step>
3. <step>

## Environment

- IronClaw version: <version or "unknown">
- Skill version: <version>
- Platform: <platform>

## Chat History

<relevant conversation excerpt, verbatim>

---
*Generated by bug-report skill v1.0.0*
```

### Step 5: Confirm

Tell the user:

> Bug report saved to `bug-reports/<filename>.md`.
> Share this file with the IronClaw developers.

## Edge Cases

- **User is vague:** Ask "Can you describe what happened?" — don't create a report without understanding the problem.
- **No clear error:** Still create the report — the chat history is valuable even without a stack trace.
- **Multiple issues:** Create separate reports for each distinct problem.
- **Sensitive content in chat:** Redact API keys, tokens, passwords. Note `[REDACTED]` in place.
